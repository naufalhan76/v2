## Findings

### F-25 [CRITICAL] — /api/cleanup-orphaned-users: No authentication whatsoever
- **File:** `src/app/api/cleanup-orphaned-users/route.ts`
- **Issue:** The endpoint is live and unauthenticated. The comment says "Add authentication check before using in production!" and "DELETE THIS FILE AFTER CLEANUP IS DONE!" — neither was done. Any anonymous caller can trigger `cleanupOrphanedAuthUsers()` which deletes auth users from Supabase.
- **Impact:** Destructive operation (user deletion) exposed to the public internet with no auth.
- **Fix:** Delete the file immediately. If still needed, gate behind `requireAuth` + SUPERADMIN role check.

### F-02 [CRITICAL] — /api/auth/api-key: Fake verification, always returns SUPERADMIN
- **File:** `src/app/api/auth/api-key/route.ts:34-40`
- **Issue:** The handler does NOT verify the API key against any database or HMAC. It only checks format (`sk_` prefix + length 67), then unconditionally returns `role: 'SUPERADMIN'` and `apiKeyValid: true`. Any string matching the format is accepted as a valid SUPERADMIN key.
- **Impact:** Authentication bypass — attacker can forge any `sk_` + 64-char string and receive a SUPERADMIN confirmation.
- **Fix:** Implement real HMAC verification against stored key hashes (the comment in `auth.ts:51-53` already documents this debt). Until then, disable or remove the endpoint.

### F-08 [CRITICAL] — /api/customers/[id]: No authentication on DELETE and PUT
- **File:** `src/app/api/customers/[id]/route.ts`
- **Issue:** Neither `DELETE` nor `PUT` handler calls `requireAuth`, `verifyAuth`, or any auth middleware. Any unauthenticated caller can delete or overwrite any customer record.
- **Impact:** Unauthenticated data destruction and modification of customer PII.
- **Fix:** Add `requireAuth(request)` guard at the top of both handlers, return 401 if null.


### F-09 [HIGH] — /api/customers/[id]: Hard delete, no soft-delete, no RBAC
- **File:** `src/app/api/customers/[id]/route.ts:13-17`
- **Issue:** Uses `.delete()` directly — violates the project convention of soft deletes (CLAUDE.md: "records are never hard-deleted"). No role check; any authenticated user (once F-08 is fixed) can delete any customer.
- **Fix:** Replace `.delete()` with `.update({ deleted_at: new Date().toISOString() })`. Add ADMIN/SUPERADMIN role gate.

### F-09b [HIGH] — /api/customers/[id] PUT: No input validation (Zod)
- **File:** `src/app/api/customers/[id]/route.ts:35-56`
- **Issue:** Body fields are passed directly to `.update()` with no Zod schema. Arbitrary keys from the request body are spread into the DB update.
- **Fix:** Validate with `UpdateCustomerSchema` (already defined in `src/app/api/schemas/index.ts:145`).

### F-07 [HIGH] — /api/customers GET: Unauthenticated access allowed
- **File:** `src/app/api/customers/route.ts:27-29`
- **Issue:** Comment says "optional for MVP - allow anon access". `requireAuth` is called but the result is never checked — a null user proceeds normally. Any anonymous caller can list all customers including PII (name, phone, email, address).
- **Fix:** Add `if (!user) return jsonError('Unauthorized', 401)` after the `requireAuth` call, or remove the comment and enforce auth consistently.

### F-03 [HIGH] — /api/auth/api-key: No rate limiting on format-check endpoint
- **File:** `src/app/api/auth/api-key/route.ts`
- **Issue:** Combined with F-02, an attacker can brute-force the 64-char suffix space. No rate limiting exists anywhere in the codebase.
- **Fix:** Resolve F-02 first (real HMAC). Then add rate limiting (e.g. upstash/ratelimit or middleware).

### F-14 [HIGH] — /api/invoices/send-email: Double auth check, wrong 403 on unauthenticated
- **File:** `src/app/api/invoices/send-email/route.ts:14-36`
- **Issue:** `requireFinanceRoleAPI` is called first (returns 403 on failure), then `supabase.auth.getUser()` is called again redundantly. If the first guard passes, the second check is dead code but adds latency. More critically, `requireFinanceRoleAPI` returns 403 (Forbidden) even for completely unauthenticated requests — should be 401 for missing auth, 403 for wrong role.
- **Fix:** Remove the redundant second auth check. Fix `requireFinanceRoleAPI` to return 401 when no user is present, 403 when role is insufficient.

### F-12 [HIGH] — /api/invoices GET: Non-standard response shape
- **File:** `src/app/api/invoices/route.ts:28-44`
- **Issue:** Returns `{ data: [...] }` without the `success` field. Error responses return `{ error: message }` without `success: false`. Breaks the documented API contract `{ success, data?, error?, pagination? }`.
- **Fix:** Use `jsonSuccess()` / `jsonError()` helpers from `src/app/api/utils.ts`.

### F-13 [MEDIUM] — /api/invoices GET: No pagination on full list
- **File:** `src/app/api/invoices/route.ts:34-44`
- **Issue:** When no `orderId` is provided, returns ALL invoices with `.select('*')` and no `range()` or `limit()`. On large datasets this will cause memory pressure and slow responses.
- **Fix:** Add `page`/`limit` query params with Zod validation and `.range()` on the Supabase query.


### F-15 [MEDIUM] — /api/invoices/send-email: Resend error details leaked to client
- **File:** `src/app/api/invoices/send-email/route.ts:352-356`
- **Issue:** `details: emailError` is returned directly in the response body. Resend error objects may contain internal API details.
- **Fix:** Log the full error server-side, return only a generic message to the client.

### F-16 [MEDIUM] — /api/invoices/send-email: No rate limiting on email send
- **File:** `src/app/api/invoices/send-email/route.ts`
- **Issue:** No rate limiting. An authenticated FINANCE user can spam-send emails to customers indefinitely.
- **Fix:** Add per-invoice or per-user rate limiting (e.g. max 5 sends per invoice per hour).

### F-01 [MEDIUM] — /api/auth/login: No rate limiting on login endpoint
- **File:** `src/app/api/auth/login/route.ts`
- **Issue:** No brute-force protection. Unlimited password attempts allowed.
- **Fix:** Add rate limiting middleware (IP-based, e.g. 10 attempts per minute).

### F-04 [MEDIUM] — Multiple routes: No RBAC beyond authentication
- **Files:** `/api/orders`, `/api/orders/create`, `/api/orders/[id]/status`, `/api/technicians`, `/api/dashboard/kpi`
- **Issue:** Any authenticated user (including TECHNICIAN role) can call these admin/dashboard endpoints. A technician JWT grants full access to order creation, status updates, and KPI data.
- **Fix:** Add role checks using `checkRole(request, ['ADMIN','SUPERADMIN'])` for admin-only operations.

### F-05 [MEDIUM] — /api/orders/create: items array validated but silently ignored
- **File:** `src/app/api/orders/create/route.ts:56-65`
- **Issue:** `items` is validated in `CreateOrderSchema` and destructured, but never passed to `createOrder()`. The audit log records `itemCount` but the items themselves are dropped.
- **Fix:** Pass `items` to `createOrder()` or remove it from the schema if intentionally unused.

### F-06 [MEDIUM] — /api/orders/[id]/status: No role gate on status transitions
- **File:** `src/app/api/orders/[id]/status/route.ts`
- **Issue:** Any authenticated user can transition any order to any status including PAID, CANCELLED, CLOSED. The `updateOrderStatus` action is called with `asAdmin: true` unconditionally.
- **Fix:** Gate destructive transitions (CANCELLED, PAID, CLOSED) behind ADMIN/SUPERADMIN role.

### F-10 [MEDIUM] — /api/ac-units: location_id filter parsed but not forwarded
- **File:** `src/app/api/ac-units/route.ts:71-75`
- **Issue:** `location_id` is validated in `GetAcUnitsQuerySchema` but not passed to `getAcUnits()`. The filter is silently dropped.
- **Fix:** Pass `location_id: filters.location_id` to `getAcUnits()`.

### F-11 [MEDIUM] — /api/ac-units/[id] PUT: No Zod validation on update body
- **File:** `src/app/api/ac-units/[id]/route.ts:79-98`
- **Issue:** Raw `body` fields are passed directly to `updateAcUnit()` with only a manual `Number()` cast on `capacity_btu`. No schema validation — arbitrary fields accepted.
- **Fix:** Validate with a Zod schema (partial version of `CreateAcUnitSchema`).


### F-17 [MEDIUM] — /api/service-reports GET: Cookie-only auth, no Bearer support
- **File:** `src/app/api/service-reports/route.ts`
- **Issue:** Uses `supabase.auth.getUser()` (cookie session) with no Bearer token fallback. Mobile/API clients using Bearer JWT get 401. Inconsistent with other routes that support both.
- **Fix:** Use `requireAuth(request)` which supports Bearer, or use the same dual-path pattern as `authenticateTechnician`.

### F-18 [MEDIUM] — /api/service-reports/[reportId]/signature: No ownership check
- **File:** `src/app/api/service-reports/[reportId]/signature/route.ts`
- **Issue:** Comment says "RLS on service_reports already gates SELECT" — but this relies entirely on RLS being correctly configured. If RLS is misconfigured or the admin client is used elsewhere, any authenticated user can fetch any signature URL. No explicit ownership check in the route handler.
- **Fix:** Add an explicit query to verify the report belongs to the requesting user or their order, rather than relying solely on RLS as the only gate.

### F-19 [MEDIUM] — /api/service-records/[id]/complete: Stub implementation, always succeeds
- **File:** `src/app/api/service-records/[id]/complete/route.ts:56-65`
- **Issue:** The handler has a `TODO` comment and returns a hardcoded success response without touching the database. Any POST to this endpoint returns 200 with fake data regardless of whether the service record exists.
- **Fix:** Implement the actual DB update or remove the endpoint until it is ready.

### F-20 [LOW] — /api/admin/reminders/run: timingSafeEqual length-leak
- **File:** `src/app/api/admin/reminders/run/route.ts:117-123`
- **Issue:** The custom `timingSafeEqual` returns `false` immediately when lengths differ, leaking the length of `CRON_SECRET` via timing. Node.js `crypto.timingSafeEqual` requires equal-length buffers but can be used with padding.
- **Fix:** Use `crypto.timingSafeEqual(Buffer.from(a.padEnd(256)), Buffer.from(b.padEnd(256)))` or compare HMAC digests of both strings.

### F-21 [LOW] — /api/technician/jobs/today: UTC date range ignores client timezone
- **File:** `src/app/api/technician/jobs/today/route.ts:21-24`
- **Issue:** "Today" is computed in server UTC. A technician in UTC+7 at 23:00 local time (16:00 UTC) will see tomorrow's jobs cut off. Comment acknowledges this but does not fix it.
- **Fix:** Accept an optional `date` query param (YYYY-MM-DD) from the client so the app can pass the local date.

### F-22 [LOW] — /api/technician/jobs/[id]/report: No transaction — partial failure possible
- **File:** `src/app/api/technician/jobs/[id]/report/route.ts:106-170`
- **Issue:** Report insert, AC unit update, order status update, and transition log are four separate DB calls with no transaction. If the order status update fails after the report is inserted, the DB is left in an inconsistent state (report exists but order still IN_PROGRESS).
- **Fix:** Wrap in a Supabase RPC/transaction or use a database function for atomicity.

### F-23 [LOW] — /api/technician/history: Status filter not validated
- **File:** `src/app/api/technician/history/route.ts:72-74`
- **Issue:** `statusFilter` is split on commas and passed directly to `.in('status', statuses)` with no validation. Arbitrary strings are sent to the DB query.
- **Fix:** Validate against the canonical status enum before using in the query.

### F-24 [LOW] — /api/technician/push/public-key: Missing success wrapper
- **File:** `src/app/api/technician/push/public-key/route.ts:12`
- **Issue:** Returns `{ publicKey }` directly via `NextResponse.json()` instead of the standard `{ success: true, data: { publicKey } }` shape. Breaks response shape consistency.
- **Fix:** Use `jsonSuccess({ publicKey })`.

### F-25b [LOW] — No health check endpoint
- **Issue:** No `/api/health` or `/api/ping` endpoint exists. Uptime monitors and load balancers have no lightweight target to probe.
- **Fix:** Add `src/app/api/health/route.ts` returning `{ success: true, data: { status: 'ok', ts: Date.now() } }` with no auth.

### F-26 [INFO] — No CORS headers on any route
- **Issue:** No `Access-Control-Allow-Origin` headers are set. Cross-origin requests from the mobile PWA or third-party integrations will be blocked by browsers.
- **Fix:** Add CORS middleware in `next.config.js` or a global middleware file for API routes that need cross-origin access.

### F-27 [INFO] — No rate limiting anywhere in the codebase
- **Issue:** Zero rate limiting on any endpoint. Applies globally — login, email send, reminder run, order creation, etc.
- **Fix:** Add rate limiting middleware (e.g. `@upstash/ratelimit` with Redis, or `next-rate-limit`) at the Next.js middleware layer.

## Notes

- **No test framework** is configured (per CLAUDE.md). All findings are from static code review only — no live request testing was performed.
- **RLS** is relied upon as a secondary gate in several routes. RLS configuration was not audited here; a separate DB-level audit is recommended.
- **Soft-delete convention** (CLAUDE.md) is violated by `DELETE /api/customers/[id]` and should be checked across all other delete operations.
- **Bearer-only vs cookie+Bearer inconsistency**: Some routes use `requireAuth` (Bearer only), others use the dual-path pattern in `authenticateTechnician` / `requireFinanceRoleAPI`. This should be standardised.
