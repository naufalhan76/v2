## Findings

### CRITICAL

#### C1 — Open self-registration endpoint (no invite gate)
**File:** `src/app/(auth)/login/page.tsx:156-247`
**Detail:** The Register tab calls `supabase.auth.signUp()` directly from the browser with no server-side guard. Any anonymous visitor can create an `auth.users` row. The toast message even advertises "You'll be assigned ADMIN role by default." If the DB trigger that creates the `user_management` row assigns ADMIN by default, an attacker can self-register and immediately gain ADMIN access.
**Impact:** Full admin privilege escalation by any internet user.
**Fix:** Remove the Register tab from the public login page entirely. Provision users only via the SUPERADMIN user-management UI (which already exists). If self-registration must remain, gate it server-side with an invite token and set `is_active = false` pending approval.

#### C2 — API key system is stateless but unverifiable — effectively broken
**File:** `src/lib/api-key.ts:25-58`, `src/app/api/middleware/auth.ts:51-54`
**Detail:** `generateApiKey()` produces `sk_<payloadHash><signature>` where both halves are truncated SHA-512 hashes of the payload. The payload is never stored. `verifyApiKey()` without a `metadataLookup` always returns `null` (line 94). The comment in `auth.ts` confirms the legacy path was a critical auth bypass and was removed — but `generateApiKey` is still callable and keys are still issued to users (see `api-keys-management.tsx`). Issued keys are permanently unverifiable: they cannot be validated, cannot be revoked, and cannot expire. Any code path that tries to authenticate with an issued key will silently fail.
**Impact:** API key feature is non-functional. Users who generate keys and share them with external clients will get silent 401s with no diagnostic path.
**Fix:** Store key metadata (hash, userId, role, exp) in a `api_keys` table on generation. On verify, look up by hash, check expiry, check `is_revoked`. Only then can HMAC verification be meaningful.


### HIGH

#### H1 — Middleware in-memory cache survives across requests in the same worker — no invalidation on logout
**File:** `src/middleware.ts:7-33`
**Detail:** `userCache` is a module-level `Map`. Once a user's `{ is_active, role }` is cached for 30 seconds, a logout or role change in the DB has no effect until the TTL expires. An admin who deactivates a user (`is_active = false`) will not see the middleware block that user for up to 30 seconds. In a serverless/edge environment the cache may also be shared across concurrent requests in the same worker instance.
**Impact:** Deactivated or demoted users retain access for up to 30 s after the DB change.
**Fix:** On logout, call a server action that sets a short-lived cookie or header to bust the cache entry. Alternatively reduce TTL to 5 s or remove the cache and rely on Supabase's own JWT caching.

#### H2 — `getUserRole()` uses `.single()` — throws on missing `user_management` row
**File:** `src/lib/auth.ts:46-57`
**Detail:** `.single()` throws a `PGRST116` error when no row is found. The error is caught and `null` is returned — but this means a user who exists in `auth.users` but has no `user_management` row silently gets `role = null`. The middleware uses `.maybeSingle()` (correct), but `getUserRole()` in `auth.ts` uses `.single()`. Any server action that calls `getUserRole()` for such a user will log an error and return null, which may cause confusing 403s rather than a clear "account not provisioned" message.
**Impact:** Inconsistent behaviour between middleware and server actions for orphaned auth users.
**Fix:** Change `.single()` to `.maybeSingle()` in `auth.ts:50`.

#### H3 — Login page role-gate only blocks TECHNICIAN and FINANCE at the client — middleware does not enforce it
**File:** `src/app/(auth)/login/page.tsx:120-122`
**Detail:** After successful Supabase auth, the login page checks `if (!['SUPERADMIN', 'ADMIN'].includes(userData.role))` and throws a client-side error. However the middleware only redirects TECHNICIAN away from `/dashboard`; it does not block FINANCE from `/dashboard`. A FINANCE user who bypasses the login page JS (e.g. via direct cookie injection or a different client) will land on `/dashboard` with no middleware block. The only protection is RLS — which is correct but the defence-in-depth layer is missing.
**Impact:** FINANCE users can access the full dashboard UI; only DB-level RLS prevents data leakage.
**Fix:** Add a middleware check: if `userRole === 'FINANCE'` and `pathname.startsWith('/dashboard/manajemen/user')` or other FINANCE-restricted paths, redirect. Also consider whether FINANCE should be blocked from the dashboard root entirely.

#### H4 — `requireFinanceRoleAPI` returns HTTP 403 for unauthenticated requests instead of 401
**File:** `src/app/api/middleware/auth.ts:130-134`
**Detail:** When no user is resolved (missing/invalid token), the function returns `errorResponse('Unauthorized: Finance role required')` with status **403**. The correct HTTP semantic for "no credentials" is 401; 403 is for "authenticated but forbidden". This breaks standard API client retry logic (401 triggers token refresh; 403 does not).
**Impact:** API clients cannot distinguish "not logged in" from "wrong role" — breaks token-refresh flows.
**Fix:** Return 401 when `user` is null, 403 only when user exists but lacks the required role.


### MEDIUM

#### M1 — Duplicate `createClient` factory in `src/lib/auth.ts`
**File:** `src/lib/auth.ts:5-30`, `src/lib/supabase-server.ts:4-29`
**Detail:** `auth.ts` defines its own `createClient()` that is identical to the one in `supabase-server.ts`. Both create a `@supabase/ssr` server client with cookie handling. `rbac.ts` imports from `supabase-server.ts`; `auth.ts` uses its own copy. Two divergent implementations of the same factory increase maintenance surface and risk subtle drift.
**Fix:** Delete the `createClient` in `auth.ts` and import from `@/lib/supabase-server` instead.

#### M2 — No TanStack Query cache clear on logout
**File:** `src/components/sidebar.tsx:269-274`, `src/components/technician/profile-content.tsx:137-146`
**Detail:** Both logout handlers call `supabase.auth.signOut()` then navigate away, but never call `queryClient.clear()` or `queryClient.removeQueries()`. Stale cached data (orders, invoices, user info) remains in memory. If a second user logs in on the same browser tab in the same session, they may briefly see the previous user's data before queries refetch.
**Fix:** Import `useQueryClient` in both logout handlers and call `queryClient.clear()` before navigating to `/login`.

#### M3 — `hasAccess()` hierarchy treats TECHNICIAN and FINANCE as equal (both level 2)
**File:** `src/lib/rbac.ts:44-51`
**Detail:** The role hierarchy assigns both TECHNICIAN and FINANCE a weight of 2. `hasAccess(userRole, 'TECHNICIAN')` returns `true` for FINANCE and vice versa. This means `hasAccess('FINANCE', 'TECHNICIAN')` is `true`, which is semantically wrong — a FINANCE user should not be considered to "have access" to technician-level operations.
**Fix:** Assign distinct weights, e.g. `TECHNICIAN: 1, FINANCE: 2` (or vice versa depending on intended hierarchy), and audit all `hasAccess()` call sites.

#### M4 — `current_user_role()` DB function has no `SET search_path` — schema injection risk
**File:** `supabase/migrations/01_v2_rls.sql:7-16`
**Detail:** The `SECURITY DEFINER` function `current_user_role()` does not set `search_path = public, pg_temp`. Without a fixed search path, a malicious user who can create objects in a schema earlier in the search path could shadow `public.user_management` and cause the function to return an arbitrary role.
**Fix:** Add `SET search_path = public, pg_temp` to both `current_user_role()` and `current_technician_id()`.

#### M5 — `API_KEY_SECRET` falls back to a hardcoded default in production
**File:** `src/lib/api-key.ts:10`
**Detail:** `const API_KEY_SECRET = process.env.API_KEY_SECRET || 'your-secret-key-change-in-production'`. If `API_KEY_SECRET` is not set in the environment, HMAC signatures are computed with a well-known string. Any attacker who reads the source (public repo or leaked bundle) can forge valid-looking API key signatures.
**Fix:** Remove the fallback. Throw an error at startup if `API_KEY_SECRET` is missing: `if (!process.env.API_KEY_SECRET) throw new Error('API_KEY_SECRET env var is required')`.


### LOW

#### L1 — Middleware `signOut()` in Edge context may not clear cookies reliably
**File:** `src/middleware.ts:116`
**Detail:** When a user is found inactive, the middleware calls `await supabase.auth.signOut()` on the SSR client. In Next.js Edge middleware, cookie mutations via the SSR client are written to the `res` object that was already constructed before the `signOut` call. The subsequent `NextResponse.redirect()` is a new response object and may not carry the cleared auth cookies, leaving the session cookie intact in the browser.
**Fix:** Instead of calling `signOut()`, manually delete the Supabase auth cookies (`sb-*-auth-token`) on the redirect response using `response.cookies.delete()`.

#### L2 — `supabase-browser.ts` guard throws at module evaluation time in SSR
**File:** `src/lib/supabase-browser.ts:5-7`
**Detail:** The `typeof window === 'undefined'` check throws an `Error` if the module is imported in a server context. Next.js App Router can accidentally import client modules during RSC evaluation. The throw will surface as an unhandled error rather than a clear import boundary violation.
**Fix:** Add `'use client'` directive at the top of `supabase-browser.ts` to make the boundary explicit and prevent accidental server-side imports.

#### L3 — `verifyAuth()` and `getUserFromRequest()` are near-identical — dead code risk
**File:** `src/app/api/middleware/auth.ts:21-44`, `55-74`
**Detail:** Both functions extract a Bearer token and call `supabase.auth.getUser(token)`. The only difference is that `verifyAuth` returns `unauthorizedResponse()` on failure while `getUserFromRequest` returns `null`. `requireAuth` (line 102) is just an alias for `getUserFromRequest`. Three functions doing the same thing increases maintenance surface.
**Fix:** Consolidate into one function that returns `{ user, error }` and let callers decide the response shape.

#### L4 — VAPID subject in `.env.example` uses a placeholder domain
**File:** `.env.example:14`
**Detail:** `VAPID_SUBJECT=mailto:admin@msn-erp.example` — `.example` is an IANA reserved TLD. If copied verbatim to production, some push services may reject the VAPID subject. Not a security issue but will cause silent push failures.
**Fix:** Update the example to use the actual production domain or a clear `REPLACE_ME` placeholder with a comment.


### INFO

#### I1 — Middleware correctly uses `getUser()` not `getSession()`
**File:** `src/middleware.ts:74`
**Detail:** `supabase.auth.getUser()` validates the JWT server-side on every request. This is the correct pattern — `getSession()` only reads the local cookie without server validation and is vulnerable to cookie tampering.
**Status:** Pass.

#### I2 — `createAdminClient()` is only called from server-only files
**File:** `src/lib/supabase-admin.ts`
**Detail:** All call sites (`actions/orders.ts`, `actions/users.ts`, `actions/profile.ts`, `actions/invoices.ts`, `lib/server/push-sender.ts`) are server actions or files marked `import 'server-only'`. No client component imports `createAdminClient`. The `SUPABASE_SERVICE_ROLE_KEY` is not prefixed `NEXT_PUBLIC_` so it cannot leak into the browser bundle.
**Status:** Pass.

#### I3 — RLS is enabled on all tables including new Phase 5 tables
**File:** `supabase/migrations/01_v2_rls.sql`, `02_phase5_reminders.sql`
**Detail:** Every table in both migrations has `ENABLE ROW LEVEL SECURITY` and at least one policy. `customer_reminders` and `reminder_rules` are covered. Anonymous users have no policies and will receive 0 rows / permission denied on all tables.
**Status:** Pass.

#### I4 — Push sender is correctly guarded with `import 'server-only'`
**File:** `src/lib/server/push-sender.ts:13`
**Detail:** The `server-only` import ensures the VAPID private key and admin client never reach the client bundle. Build will fail if this module is accidentally imported in a client component.
**Status:** Pass.

#### I5 — DISPATCHER role (V1 legacy) falls through to `null` in `getUserRole()`
**File:** `src/lib/auth.ts:41-57`, `src/lib/rbac.ts:5`
**Detail:** The `UserRole` type only includes `SUPERADMIN | ADMIN | TECHNICIAN | FINANCE`. If a legacy V1 user has `role = 'DISPATCHER'` in `user_management`, `getUserRole()` returns the string `'DISPATCHER'` but TypeScript treats it as `string`, not `UserRole`. All `hasAccess()` calls will return `false` (no match in hierarchy map). The middleware will not redirect them to `/technician` (role check is `=== 'TECHNICIAN'`), so they land on `/dashboard` but all server actions will deny them. Effectively a soft-lock.
**Recommendation:** Add a migration to remap legacy DISPATCHER rows to ADMIN or TECHNICIAN as appropriate, or add explicit handling in `getUserRole()`.
