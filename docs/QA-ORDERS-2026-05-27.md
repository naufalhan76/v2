## CRITICAL Findings

### [CRITICAL] createOrder() inserts legacy status NEW — runtime crash
File: src/lib/actions/orders.ts:201
Test: INSERT INTO orders ... status = 'NEW'
Expected: Order created with status PENDING
Actual: ERROR: invalid input value for enum order_status: "NEW" — every call to createOrder() will throw and return success:false
Fix suggestion: Change line 201 from `status: 'NEW'` to `status: 'PENDING'`

### [CRITICAL] createOrderWithItems() inserts legacy status ACCEPTED — runtime crash
File: src/lib/actions/create-order.ts:260
Test: INSERT INTO orders ... status = 'ACCEPTED'
Expected: Order created with status PENDING (no tech) or ASSIGNED (with tech)
Actual: ERROR: invalid input value for enum order_status: "ACCEPTED" — every order creation without a technician will fail
Fix suggestion: Change line 260 from `'ACCEPTED'` to `'PENDING'`

### [CRITICAL] updateOrderStatus() has no state machine enforcement
File: src/lib/actions/orders.ts:225-299
Test: UPDATE orders SET status = 'ASSIGNED' WHERE order_id = 'TEST-ORDER-C' (was COMPLETED); UPDATE orders SET status = 'PENDING' WHERE order_id = 'TEST-ORDER-D' (was PAID)
Expected: Illegal transitions rejected
Actual: Both updates succeeded — DB has no transition guards, and updateOrderStatus() never calls canTransition(). Any status can be set to any other status by any caller.
Fix suggestion: Add canTransition() check at top of updateOrderStatus() using the caller role; reject with error if invalid


## HIGH Findings

### [HIGH] cancelOrder() allows cancelling PAID orders — no guard
File: src/lib/actions/orders.ts:482-564
Test: Set order to PAID, then call cancelOrder() — UPDATE orders SET status = 'CANCELLED' succeeded
Expected: PAID orders are terminal and cannot be cancelled
Actual: cancelOrder() fetches current status but never checks it before proceeding. PAID -> CANCELLED transition succeeds silently.
Fix suggestion: After fetching currentOrder, add: `if (currentOrder.status === 'PAID') return { success: false, error: 'Cannot cancel a paid order' }`

### [HIGH] order_technicians has no UNIQUE constraint — duplicate leads and helpers allowed
File: src/lib/actions/orders.ts:420-450 (addHelperTechnician)
Test: INSERT two rows with same (order_id, technician_id, role='lead') — both inserted. INSERT two rows with same (order_id, technician_id, role='helper') — both inserted.
Expected: Duplicate assignments rejected at DB level
Actual: No UNIQUE constraint on (order_id, technician_id, role). Multiple lead rows per order are possible. removeHelperTechnician() deletes ALL matching rows when duplicates exist, silently over-deleting.
Fix suggestion: Add UNIQUE constraint: ALTER TABLE order_technicians ADD CONSTRAINT uq_order_tech_role UNIQUE (order_id, technician_id, role)

### [HIGH] cancelOrder() uses invalid ac_status enum value PENDING
File: src/lib/actions/orders.ts:517
Test: UPDATE ac_units SET status = 'PENDING' — ERROR: invalid input value for enum ac_status: "PENDING"
Expected: AC units with PENDING status reset to INACTIVE on order cancel
Actual: ac_status enum is {ACTIVE, INACTIVE, RETIRED} — PENDING does not exist. The .eq('status', 'PENDING') filter will throw a runtime error, though cancelOrder() catches it and continues (line 519-522).
Fix suggestion: Remove the .eq('status', 'PENDING') guard or use a valid status value. Clarify intended AC unit lifecycle with correct enum values.

### [HIGH] dashboard.ts KPI queries use legacy status values that cannot exist in DB
File: src/lib/actions/dashboard.ts:64, 72, 126, 133
Test: SELECT COUNT(*) FROM orders WHERE status IN ('NEW','ACCEPTED','EN ROUTE','ARRIVED') — ERROR: invalid input value for enum order_status
Expected: pendingOrders KPI counts all active orders; completedOrders counts PAID orders
Actual: Queries include NEW, ACCEPTED, EN ROUTE, ARRIVED, CLOSED which are not in the order_status enum. Supabase PostgREST may silently ignore invalid enum values or error. pendingOrders undercounts (misses EN_ROUTE, IN_PROGRESS). completedOrders misses INVOICED.
Fix suggestion: Update pendingOrders to IN ('PENDING','ASSIGNED','EN_ROUTE','IN_PROGRESS') and completedOrders to IN ('COMPLETED','INVOICED','PAID')

### [HIGH] deleteOrder() performs hard delete — violates soft-delete convention
File: src/lib/actions/orders.ts:566-590
Test: Static analysis — function issues DELETE FROM orders
Expected: Per CLAUDE.md: "Soft deletes — records are never hard-deleted"
Actual: deleteOrder() issues a hard DELETE with no soft-delete flag, permanently removing the order and cascading to order_items, order_technicians, order_status_transitions
Fix suggestion: Replace with UPDATE orders SET deleted_at = now() and add deleted_at IS NULL filter to all queries


## MEDIUM Findings

### [MEDIUM] getOrders() dateFrom/dateTo filters on created_at not scheduled_visit_date
File: src/lib/actions/orders.ts:80-90
Test: Static analysis — filters use .gte('created_at') and .lte('created_at')
Expected: Date range filter applies to the visit/scheduled date for operational queries
Actual: Filters apply to created_at (order creation time), not scheduled_visit_date. Searching "orders for next week" returns wrong results.
Fix suggestion: Add a separate scheduledDateFrom/scheduledDateTo filter pair that queries scheduled_visit_date, or document that dateFrom/dateTo is creation-date only

### [MEDIUM] getOrders() applies dateTo filter twice — duplicate clause
File: src/lib/actions/orders.ts:84-90
Test: Static analysis — lines 84-86 and 88-90 are identical .lte('created_at', dateTo) blocks
Expected: Single filter application
Actual: dateTo filter applied twice. Harmless (same result) but indicates copy-paste error and wastes a query condition slot.
Fix suggestion: Remove the duplicate block at lines 88-90

### [MEDIUM] createOrderWithItems() technician assignment failure leaves order in ASSIGNED state with no technician
File: src/lib/actions/create-order.ts:418-425
Test: Static analysis — on order_technicians insert error, code logs but does not rollback
Expected: Atomic operation — order + items + technician assignments all succeed or all fail
Actual: If technician insert fails, order exists in ASSIGNED status with assigned_technician_id set but no row in order_technicians. Order appears assigned but has no technician record.
Fix suggestion: Either throw on techError to trigger the outer catch (which should then rollback order + AC units), or wrap the entire operation in a Postgres transaction via RPC


## LOW Findings

### [LOW] rescheduleOrder() accepts empty reason string
File: src/lib/actions/orders.ts:592-666
Test: rescheduleOrder({ orderId, reason: '', newScheduledDate }) — transition logged as "Reschedule: "
Expected: Reason is required and validated before proceeding
Actual: Empty string reason is accepted and stored. Audit trail entry is meaningless.
Fix suggestion: Add validation: `if (!params.reason?.trim()) return { success: false, error: 'Reschedule reason is required' }`

### [LOW] orders table has no notes column — createOrderWithItems() silently drops notes
File: src/lib/actions/create-order.ts:291
Test: SELECT column_name FROM information_schema.columns WHERE table_name = 'orders' AND column_name = 'notes' — 0 rows
Expected: Order notes saved to DB
Actual: orders table has no notes column. The insert at line 291 includes notes: input.notes which Supabase will silently ignore or error depending on RLS/schema settings. The description column exists but notes does not.
Fix suggestion: Either rename notes to description in the insert, or add a notes column via migration

## Pure Utility Function Tests (order-status.ts, order-utils.ts)

All pure functions tested via static analysis and logic review:

### getNextStates(status, role) — PASS
- Terminal states (PAID, CANCELLED) return [] for all roles
- TECHNICIAN cannot transition PENDING or COMPLETED
- FINANCE can only transition COMPLETED->INVOICED and INVOICED->PAID
- Legacy values (NEW, ACCEPTED) correctly map to PENDING before lookup

### canTransition(from, to, role) — PASS
- Returns false for wrong role (TECHNICIAN cannot do PENDING->ASSIGNED)
- Returns false for backward transitions not in rules
- Note: this function is never called by updateOrderStatus() — see CRITICAL finding above

### toCanonical(status) — PASS with note
- Null/undefined returns PENDING (safe fallback)
- Unknown values return PENDING (silent fallback — could mask bugs)
- All legacy values map correctly

### isTerminalState(status) — PASS
- PAID and CANCELLED return true
- All others return false

### getUrgencyLevel(order) — PASS
- Terminal orders return 'terminal'
- Null date returns 'future' (safe default)
- Invalid date string returns 'future' (safe default via isNaN check)
- Past date returns 'overdue', today returns 'today', future returns 'future'

### groupOrdersByStatus(orders) — PASS
- CANCELLED orders excluded from board (getColumnForStatus returns null)
- EN_ROUTE and IN_PROGRESS both map to ACTIVE column

### filterOrders(orders, filters) — PASS with note
- technicianId filter checks lead.technician_id but casts via (lead as unknown as {technician_id?}) — fragile cast, technician_id not in OrderForDisplay type
- All other filters work correctly

### sortOrdersByUrgency(orders) — PASS
- Correct rank order: overdue(0) < today(1) < future(2) < terminal(3)
- Secondary sort by date string (localeCompare) is correct for ISO date strings
