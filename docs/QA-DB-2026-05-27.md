**[CRITICAL] FINANCE can read technicians but with_check excludes FINANCE from writes — policy is split-brain**

Policy `technicians_admin_all` (ALL cmd):
- `qual` (SELECT/UPDATE/DELETE filter): allows SUPERADMIN, ADMIN, **FINANCE**
- `with_check` (INSERT/UPDATE check): allows SUPERADMIN, ADMIN only — **excludes FINANCE**

This means FINANCE can SELECT, DELETE technician rows but cannot INSERT or UPDATE them. The DELETE gap is the critical issue: FINANCE can delete technician records, which is almost certainly unintended.

**[CRITICAL] Same split-brain on service_records_admin_all**

Policy `service_records_admin_all` (ALL cmd):
- `qual`: allows SUPERADMIN, ADMIN, **FINANCE**
- `with_check`: allows SUPERADMIN, ADMIN only

FINANCE can DELETE service_records rows. No INSERT/UPDATE — but DELETE via the qual path on an ALL policy is a real privilege escalation risk.

**[HIGH] customer_reminders — no TECHNICIAN read policy**  
TECHNICIANs cannot read customer_reminders at all. If the app needs to show technicians their upcoming service reminders, this is a functional gap.

**[MEDIUM] reminder_rules — FINANCE has read access but no write**  
`reminder_rules_finance_read` gives FINANCE SELECT. Consistent with read-only finance access, but worth confirming this is intentional since reminder rules affect billing-adjacent workflows.

**[MEDIUM] sheet_sync_config — SUPERADMIN only (ALL)**  
No read policy for ADMIN. If ADMIN needs to view sync config this is a gap.

**[MEDIUM] service_reminders — no TECHNICIAN policy**

**[MEDIUM] orders.order_date is DATE not TIMESTAMPTZ**
Order creation time is operationally important (SLA tracking, same-day ordering). Using DATE loses the time component. The `created_at` TIMESTAMPTZ column exists and should be the canonical creation timestamp — `order_date` may be redundant or should be renamed `requested_date`.

**[MEDIUM] payment_records.payment_date and service_records.service_date are DATE**
Financial audit trails and service history benefit from full timestamps. Recommend TIMESTAMPTZ.

### TEXT vs VARCHAR consistency

**[LOW] Mixed TEXT and VARCHAR(n) usage**
Most columns use TEXT (correct for PostgreSQL — no performance difference). Some columns use VARCHAR without length limits (effectively same as TEXT) and a few use VARCHAR with implicit limits via the column definition. Examples: `customers.customer_name character varying`, `technicians.technician_name character varying`. These are functionally equivalent to TEXT in PostgreSQL but inconsistent. Recommend standardizing to TEXT for new columns.

### NUMERIC precision

**[LOW] Most NUMERIC columns have no precision/scale specified**
Columns like `invoice_items.unit_price numeric`, `order_addons.total_price numeric` use unconstrained NUMERIC. Only `service_reports.actual_total_price numeric(12,2)` has explicit precision. For currency values, recommend `NUMERIC(12,2)` consistently to prevent accidental floating-point storage and clarify intent.

### ENUM vs free-text

**[HIGH] invoices.status, invoices.payment_status, invoices.invoice_type are TEXT not ENUM**
These fields have defined value sets (DRAFT/SENT/PAID etc for status, PROFORMA/FINAL for invoice_type, UNPAID/PARTIAL/PAID for payment_status per CLAUDE.md) but are stored as unconstrained TEXT. The `payment_status` and `payment_method` ENUMs exist but are not used on the invoices table. Invalid values can be inserted without constraint.

**[MEDIUM] customer_reminders.status, service_reminders.status, invoice_communications.status are TEXT**
All have defined state machines in application code but no DB-level constraint. Recommend CHECK constraints or ENUMs.

**[MEDIUM] order_technicians.role is VARCHAR not ENUM**
Only values used are "lead" and "support" per codebase, but no DB constraint enforces this. A typo ("Lead" vs "lead") would silently break the RLS policy `orders_tech_update_own` which checks `ot.role = 'lead'`.

## 11. Recommendations Summary

### CRITICAL (fix before go-live)

**C1. FINANCE can DELETE technicians (technicians_admin_all split-brain)**
The ALL policy uses different role sets for qual vs with_check. FINANCE passes the qual filter (can SELECT, DELETE) but fails with_check (cannot INSERT, UPDATE). DELETE is not blocked by with_check — it uses qual only. Fix: split into separate SELECT and INSERT/UPDATE/DELETE policies, or change cmd to SELECT for FINANCE.

```sql
-- Fix: replace technicians_admin_all with explicit policies
DROP POLICY technicians_admin_all ON technicians;
CREATE POLICY technicians_admin_write ON technicians
  FOR ALL TO authenticated
  USING (current_user_role() = ANY (ARRAY['SUPERADMIN'::role_type, 'ADMIN'::role_type]))
  WITH CHECK (current_user_role() = ANY (ARRAY['SUPERADMIN'::role_type, 'ADMIN'::role_type]));
CREATE POLICY technicians_finance_read ON technicians
  FOR SELECT TO authenticated
  USING (current_user_role() = 'FINANCE'::role_type);
```

**C2. FINANCE can DELETE service_records (service_records_admin_all split-brain)**
Same pattern as C1. Fix:

```sql
DROP POLICY service_records_admin_all ON service_records;
CREATE POLICY service_records_admin_write ON service_records
  FOR ALL TO authenticated
  USING (current_user_role() = ANY (ARRAY['SUPERADMIN'::role_type, 'ADMIN'::role_type]))
  WITH CHECK (current_user_role() = ANY (ARRAY['SUPERADMIN'::role_type, 'ADMIN'::role_type]));
CREATE POLICY service_records_finance_read ON service_records
  FOR SELECT TO authenticated
  USING (current_user_role() = ANY (ARRAY['SUPERADMIN'::role_type, 'ADMIN'::role_type, 'FINANCE'::role_type]));
```

### HIGH (fix soon)

**H1. Missing indexes on hot FK/filter columns**
Add at minimum:

```sql
CREATE INDEX idx_locations_customer ON locations(customer_id);

```

**H2. invoices.status/payment_status/invoice_type are unconstrained TEXT**
Add CHECK constraints or migrate to ENUMs.

**H3. service_reports and service_records do not CASCADE on order delete**
Inconsistent with order_items/order_technicians. Add ON DELETE CASCADE or document the intentional difference.

**H4. order_technicians.role is VARCHAR with no constraint**
A case mismatch breaks the lead-check RLS policy silently.
```sql
ALTER TABLE order_technicians ADD CONSTRAINT order_technicians_role_check
  CHECK (role IN ('lead', 'support'));
```
